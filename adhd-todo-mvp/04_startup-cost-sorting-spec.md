## 0. 为什么这个模块是产品的灵魂

ADHD 用户的核心障碍不是"事情多"，是**"开始难"**。

神经科学解释：ADHD 的前额叶皮层多巴胺不足，导致"任务启动"（Task Initiation）这个执行功能受损。用户面对一个任务时，大脑不是按逻辑顺序走，而是按**"哪个最不费劲"**走。

传统 Todo App 按截止时间、优先级、项目分类排序——这些维度对 ADHD 用户无效，因为它们假设用户能"理性地从第一个任务开始"。

**FocusCut 的排序逻辑：按启动成本从低到高排，让用户先拿到最容易开始的那一步。**

> 类比：不是给你一张地图让你规划路线，而是直接告诉你"往前走 3 步，左转"。

## 1. 启动成本模型（Startup Cost Model）

### 1.1 定义

**启动成本（Startup Cost, SC）**= 用户从"看到这条子任务"到"开始执行第一个物理动作"所需的心理能量总和。

单位：无量纲分数，1–10 分制。

### 1.2 五维评分模型

每个子任务由 AI 在拆解时同步评估以下 5 个维度：

| 维度 | 英文 | 权重 | 评分标准（1=最低，5=最高） |
|------|------|------|---------------------------|
| **认知负荷** | Cognitive Load | 0.30 | 1=纯物理动作（"打开 Word"）；3=需要简单思考（"写第一段开头"）；5=需要深度思考（"分析数据趋势"） |
| **模糊度** | Ambiguity | 0.25 | 1=动作完全明确（"点击发送按钮"）；3=方向明确但细节待定（"写个大纲"）；5=需要自己定义做什么（"整理一下思路"） |
| **情绪阻力** | Emotional Resistance | 0.20 | 1=中性/愉快（"听首歌放松"）；3=略有压力（"回复客户邮件"）；5=高焦虑/羞耻感（"给导师发拖延了两周的消息"） |
| **上下文切换** | Context Switch | 0.15 | 1=当前工具/环境直接可做；3=需要切换 1 个工具（打开新 App）；5=需要切换环境（出门、找人） |
| **时间感知** | Time Perception | 0.10 | 1=感觉"几秒就完"；3=感觉"需要几分钟"；5=感觉"要花很久"（即使实际不长） |

### 1.3 启动成本计算公式

```
SC = 0.30 × CognitiveLoad + 0.25 × Ambiguity + 0.20 × EmotionalResistance + 0.15 × ContextSwitch + 0.10 × TimePerception
```

**示例：**

| 子任务 | 认知 | 模糊 | 情绪 | 切换 | 时间 | SC |
|--------|------|------|------|------|------|-----|
| 打开 Word 新建文档 | 1 | 1 | 1 | 2 | 1 | **1.15** |
| 写第一段开头句 | 3 | 3 | 2 | 1 | 2 | **2.45** |
| 给导师发拖延两周的消息 | 2 | 2 | 5 | 1 | 2 | **2.50** |
| 分析上季度数据趋势 | 5 | 3 | 3 | 2 | 4 | **3.65** |
| 整理一下思路 | 4 | 5 | 2 | 1 | 3 | **3.30** |

## 2. 排序算法

### 2.1 核心原则

**启动成本低的排前面，让用户"先动起来"。**

动起来之后，多巴胺分泌，后续任务的感知启动成本会自然下降（心理学：行为激活效应 Behavioral Activation）。

### 2.2 精力状态 × 排序策略矩阵

| 精力状态 | 排序策略 | 逻辑 |
|----------|----------|------|
| **低** 😴 | **严格按 SC 升序** | 用户处于"什么都难开始"状态，必须把最简单的放第一个。忽略逻辑依赖，先动起来比顺序正确更重要。 |
| **中** 🙂 | **SC 升序为主，逻辑依赖为辅** | 如果两个子任务有强依赖关系（如"写大纲"→"写正文"），保持依赖顺序；无依赖关系的按 SC 排序。 |
| **高** ⚡ | **逻辑顺序为主，SC 为辅** | 用户有足够能力处理高启动成本任务，按任务的自然逻辑顺序排列，但仍然避免把 SC≥4 的任务放在第一位。 |

### 2.3 排序算法伪代码

```python
def sort_subtasks(subtasks: List[Subtask], energy: EnergyLevel) -> List[Subtask]:
    """
    输入: AI 拆解后的子任务列表（每个已带 SC 分数）
    输出: 排序后的子任务列表
    """
    
    # Step 1: 识别依赖关系
    # AI 在拆解时标记有强依赖的对（如 task_2.depends_on = task_1.id）
    # 无标记的视为可独立执行
    
    # Step 2: 按精力状态选择策略
    if energy == 'low':
        # 严格 SC 升序，忽略依赖（用户需要先动起来）
        return sorted(subtasks, key=lambda t: t.startup_cost)
    
    elif energy == 'medium':
        # 拓扑排序 + SC 加权
        # 有依赖的保持顺序，无依赖的按 SC 排
        groups = topological_group(subtasks)  # 按依赖分组
        for group in groups:
            group.sort(key=lambda t: t.startup_cost)
        return flatten(groups)
    
    elif energy == 'high':
        # 逻辑顺序为主，但首条任务 SC 必须 < 4
        ordered = topological_sort(subtasks)
        if ordered[0].startup_cost >= 4:
            # 找到 SC < 4 的最近任务，提到最前
            for i, t in enumerate(ordered):
                if t.startup_cost < 4:
                    ordered.insert(0, ordered.pop(i))
                    break
        return ordered
```

### 2.4 硬性约束（所有精力状态下）

1. **首条子任务 SC 必须 < 4**：如果所有子任务 SC ≥ 4，AI 必须追加一条"暖身任务"（SC ≤ 2），如"打开相关文件"、"倒杯水再回来"
2. **相邻子任务 SC 跳幅 ≤ 3**：不允许从 SC=1 直接跳到 SC=5，中间需要过渡
3. **SC ≥ 4 的子任务不超过总数的 1/3**：避免用户在执行过程中遇到太多高阻力任务

## 3. AI Prompt 规格（锁定版，R03 直接用）

### 3.1 System Prompt 增量

在现有 System Prompt（02_PRD.md §4.4）基础上，追加以下指令：

```
## 启动成本评估（必须执行）

拆解子任务时，你必须同时为每个子任务评估"启动成本"（Startup Cost）。

启动成本 = 用户从"看到这条任务"到"开始动手"的心理阻力。

评估 5 个维度（每项 1-5 分）：
- cognitive_load: 认知负荷。纯动手=1，需要思考=3，需要深度分析=5
- ambiguity: 模糊度。动作完全明确=1，方向清楚但细节待定=3，需要自己定义=5
- emotional_resistance: 情绪阻力。中性=1，略有压力=3，高焦虑/回避=5
- context_switch: 上下文切换。当前环境可做=1，需换工具=3，需换环境=5
- time_perception: 时间感知。感觉几秒完=1，感觉几分钟=3，感觉很久=5

启动成本分数 = cognitive_load×0.30 + ambiguity×0.25 + emotional_resistance×0.20 + context_switch×0.15 + time_perception×0.10

然后按精力状态排序：
- 精力低：严格按启动成本从低到高排
- 精力中：有依赖关系的保持顺序，无依赖的按启动成本排
- 精力高：按逻辑顺序排，但第一条启动成本必须<4

硬性约束：
1. 第一条子任务的启动成本必须 < 4。如果所有任务都 ≥ 4，先加一条暖身任务（如"打开相关文件"）
2. 相邻任务启动成本跳幅不超过 3
3. 启动成本 ≥ 4 的任务不超过总数的 1/3
```

### 3.2 输出格式升级

```json
{
  "subtasks": [
    {
      "action": "打开 Word 新建文档",
      "estimated_minutes": 1,
      "startup_cost": {
        "cognitive_load": 1,
        "ambiguity": 1,
        "emotional_resistance": 1,
        "context_switch": 2,
        "time_perception": 1,
        "score": 1.15
      },
      "depends_on": null
    },
    {
      "action": "写第一段开头句",
      "estimated_minutes": 3,
      "startup_cost": {
        "cognitive_load": 3,
        "ambiguity": 3,
        "emotional_resistance": 2,
        "context_switch": 1,
        "time_perception": 2,
        "score": 2.45
      },
      "depends_on": 0
    }
  ],
  "warmup_task": null,
  "sort_strategy": "low_energy_strict_sc"
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `startup_cost.score` | float | ✅ | 最终启动成本分数，1-10 |
| `startup_cost.*` | int(1-5) | ✅ | 5 个维度的原始评分，用于调试和后续优化 |
| `depends_on` | int \| null | ✅ | 依赖的子任务 index，null 表示无依赖 |
| `warmup_task` | object \| null | ✅ | 当所有子任务 SC≥4 时，生成的暖身任务 |
| `sort_strategy` | string | ✅ | 实际使用的排序策略，用于前端展示和调试 |

## 4. 前端展示规格（给 R02 的设计约束）

### 4.1 启动成本对用户不可见

**关键决策：SC 分数和维度评分不展示给用户。**

原因：
- ADHD 用户看到评分会产生"我连这个都做不了"的自我否定
- 排序是隐性的，用户只感受到"这个 App 总是先给我简单的"
- 信息元素 ≤ 3 的原则（PRD §6.1）

### 4.2 暖身任务的展示

当 AI 插入暖身任务时，展示为普通子任务，但加一个微妙的视觉标记：

```
☕ 打开相关文件          ← 暖身任务，带咖啡图标
   预估 1 分钟
```

不解释"这是暖身任务"，只用 ☕ 图标暗示"轻松开始"。

### 4.3 排序策略提示（可选，Should Have）

在精力状态选择后，显示一行小字提示排序逻辑：

| 精力 | 提示文案 |
|------|----------|
| 低 | "已为你安排最轻松的开始方式 💤" |
| 中 | "先简单后复杂，帮你热身 🙂" |
| 高 | "按最佳顺序排列，冲吧 ⚡" |

## 5. 边界情况 & Fallback

### 5.1 AI 未返回 startup_cost

**Fallback**：前端按原始顺序展示，不排序。同时记录 `missing_startup_cost` 事件到指标系统。

### 5.2 AI 返回的 SC 分数全相同

**Fallback**：按原始逻辑顺序展示。如果全为 SC=5，触发暖身任务插入逻辑。

### 5.3 用户手动编辑子任务后

**规则**：用户编辑子任务内容后，该子任务的 SC 分数标记为 `stale`（过期）。

- 如果用户只改了文字描述：保留原 SC 分数（因为启动成本主要取决于任务类型，不是措辞）
- 如果用户增删了子任务：触发重新排序（前端本地排序，不调 API）

### 5.4 重新拆解（"重拆"按钮）

**规则**：重新拆解时，AI 重新评估所有子任务的 SC 分数和排序。用户之前的编辑被覆盖。

### 5.5 依赖环（A depends on B, B depends on A）

**规则**：AI 输出时不应产生依赖环。前端做一次环检测，如果检测到，忽略依赖关系，按 SC 纯排序。

## 6. 指标体系（本模块专属）

### 6.1 核心指标

| 指标 | 定义 | MVP 目标 | 用途 |
|------|------|----------|------|
| **首步启动率** | 看到第一条子任务后 30 秒内点击"搞定了"的比例 | ≥ 75% | 验证 SC 排序是否降低了启动阻力 |
| **暖身任务触发率** | AI 插入暖身任务的比例 | 监控即可 | 如果 >30%，说明 AI 拆解粒度不够细 |
| **排序跳过率** | 用户跳过第一条子任务直接看下一条的比例 | ≤ 15% | 如果高，说明第一条 SC 仍然太高 |

### 6.2 诊断指标

| 指标 | 定义 | 用途 |
|------|------|------|
| 平均首条子任务 SC | 所有任务第一条子任务的平均启动成本 | 目标 < 2.5，验证排序效果 |
| SC 分布 | 所有子任务的 SC 分数分布 | 检查是否集中在高分段（拆解不够细） |
| 依赖关系使用率 | AI 标记 depends_on 的比例 | 评估 AI 是否理解任务结构 |
| 手动重排率 | 用户在编辑阶段调整子任务顺序的比例 | 如果高，说明 AI 排序不符合用户直觉 |

## 7. 迭代路线（v2 预留）

### 7.1 个性化 SC 模型（v2）

- 记录用户对每个子任务的"实际启动时间"（从看到到点击完成）
- 用实际数据反向校准 5 个维度的权重
- 不同用户的权重不同（有人对情绪阻力敏感，有人对认知负荷敏感）

### 7.2 情绪标注（v2）

- 允许用户对子任务添加情绪标签（😰 😤 😒）
- 情绪标签直接影响 EmotionalResistance 维度评分
- 用于更精准的排序

### 7.3 任务记忆（v2）

- 如果用户之前做过类似任务，记录哪些子任务被跳过/编辑
- 下次遇到类似任务时，AI 参考历史数据调整排序
